# Agent Content Flow — End-to-End

How a blog/page published in Sanity becomes an AI-crawlable `/agents/{slug}` page on this site.

> **Audience.** Anyone on the team who needs to understand how the pieces fit together: editors, the PublishForge dev, future contributors to this repo, reviewers.

## TL;DR

When an editor publishes a blog, **two artifacts** end up in production:

| Surface       | URL              | Source of truth                                         | Audience         |
| ------------- | ---------------- | ------------------------------------------------------- | ---------------- |
| Blog post     | `/{slug}`        | Sanity document                                         | Humans           |
| Agent article | `/agents/{slug}` | Markdown file in this repo (`content/agents/{slug}.md`) | AI agents / LLMs |

The agent article is **not the blog post itself.** It is an LLM-rewritten derivative, generated by PublishForge during the publish flow, optimized for AI consumption (clearer structure, explicit facts, grounded in compiled WIKI knowledge). It lives alongside the blog post on the site, discoverable through `/llms.txt` and `/sitemap-agents.xml`, but not linked from any human-facing navigation.

---

## The two-surface model

This is the most important concept to internalize: **blog content and agent content are parallel surfaces from a shared source.**

```
                    Editor writes in Sanity Studio
                              │
                              ▼
                    ┌──────────────────────┐
                    │   Blog draft (CMS)   │
                    └──────────┬───────────┘
                               │
                Editor publishes via PublishForge
                               │
              ┌────────────────┴────────────────┐
              │                                 │
              ▼                                 ▼
   ┌──────────────────┐              ┌─────────────────────┐
   │ Blog post (live) │              │ Agent article (live)│
   │  /blog-slug      │              │  /agents/blog-slug  │
   │  (Sanity-driven, │              │  (markdown-driven,  │
   │   human-styled)  │              │   AI-targeted)      │
   └──────────────────┘              └─────────────────────┘
```

The agent article is generated by Claude (via PublishForge's agent transformation prompt) using:

- The blog content itself, plus
- WIKI articles referenced by the blog's `wiki_article_refs` field.

The transformation is deterministic in shape (validated against a Zod schema in PublishForge), but creative in language — every blog gets its own rewrite.

Typical end-to-end latency from "editor clicks Publish" to "new `/agents/{slug}` page is live": **30–90 seconds**, dominated by the LLM call (~10–30s) and the Vercel build (~30–60s).

## What each repo does

### PublishForge (other repo)

- Hosts the operator console.
- Owns Sanity writes.
- Runs the agent transformation LLM.
- Holds the GitHub App credentials.
- Commits markdown to this repo.
- Fires the Vercel deploy hook.
- Polls Vercel for deploy status.
- Tracks all of this in its own database (`publish_events` table).

### This repo (StackShift site)

- Stores agent markdown files in `content/agents/`.
- Builds three discovery artifacts at every deploy:
  - `public/llms.txt` — llmstxt.org-format index.
  - `public/sitemap-agents.xml` — XML sitemap.
  - `public/robots.txt` — declares both sitemaps.
- Renders one static HTML page per `*.md` file at `/agents/{slug}`.
- Embeds JSON-LD Schema.org markup on each page (`Article` + optional `FAQPage`).
- Does **not** touch Sanity for agent content.
- Does **not** run any LLM.
- Does **not** receive any inbound HTTP from PublishForge or AI crawlers beyond standard GETs.

### Shared infrastructure

- **GitHub App** — registered once globally by the PublishForge team; installed per-customer on each StackShift repo. Grants `Contents: Read & write` so PublishForge can commit `*.md` files.
- **Vercel deploy hook** — generated per-customer in the Vercel dashboard for this site, URL handed to the PublishForge team to store in their per-customer config.

## Editor experience

**Nothing changes for editors.** They write in Sanity Studio and click Publish in PublishForge the same way they always did. The agent layer is invisible — it just appears as a byproduct on the next deploy. No manual markdown authoring. No new workflow.

If something goes wrong with agent generation specifically (e.g., LLM schema validation fails), PublishForge records a failure on the `publish_events.agent_generation_status` field and surfaces it in the admin UI. The blog itself still publishes — agent generation is non-blocking for the blog publish.

---

## Updating an existing blog

Same flow. PublishForge regenerates the agent markdown using the new blog content, commits it to the same `content/agents/{slug}.md` path (overwriting), Vercel rebuilds, the agent page updates.

The slug is **stable** across updates — the URL never changes. External AI references to `/agents/{slug}` remain valid even when the blog body is edited.

## What's stored where (mental model)

| Data                               | Lives in                                                         | Stable across updates?           |
| ---------------------------------- | ---------------------------------------------------------------- | -------------------------------- |
| Blog draft + published copy        | Sanity                                                           | Yes (Sanity ID); slug may change |
| WIKI articles (compiled knowledge) | PublishForge Supabase                                            | Yes (article ID)                 |
| Agent markdown file                | This repo, `content/agents/{slug}.md`                            | Yes (slug = filename)            |
| llms.txt entry                     | Regenerated at every build from the markdown files               | Reflects current state           |
| sitemap-agents.xml entry           | Regenerated at every build                                       | Reflects current state           |
| Page rendered HTML                 | `.next/server/pages/en/agents/{slug}.html` (Vercel build output) | Regenerated at every build       |

## Failure modes and where to look

| Symptom                                                        | Likely cause                                                           | First place to look                                     |
| -------------------------------------------------------------- | ---------------------------------------------------------------------- | ------------------------------------------------------- |
| Editor publishes, no `/agents/{slug}` page appears             | LLM transformation failed; PublishForge skipped the commit             | PublishForge `publish_events.agent_generation_status`   |
| Markdown committed but page isn't live                         | Vercel deploy hook not fired; deploy failed                            | Vercel deployment dashboard                             |
| Page exists but JSON-LD missing                                | Frontmatter parsing failure (malformed YAML)                           | View page source; check the markdown frontmatter syntax |
| Page exists but missing FAQ schema                             | Frontmatter doesn't have `faq` array (correct behavior)                | Confirm the source blog actually had FAQs to extract    |
| `/llms.txt` doesn't list a known page                          | `prebuild` didn't run, OR the markdown file isn't in `content/agents/` | Vercel build logs — look for `[generate-llms-txt]` line |
| `/llms.txt` empty in production                                | No content has been committed yet; expected for fresh deploys          | Wait for first PublishForge publish                     |
| Old URLs in `llms.txt` / `sitemap-agents.xml` after env change | `NEXT_PUBLIC_SITE_URL` not set in Vercel production env                | Vercel project Environment Variables                    |

## Where to extend in Sprint 2

This is what makes the architecture compose well. None of these require touching the rendering layer:

| Sprint 2 work                                          | Where it goes                                                          |
| ------------------------------------------------------ | ---------------------------------------------------------------------- |
| PIM-derived product agent pages                        | Append `"content/agents-products"` to `lib/agents/content-dirs.ts`     |
| Product / Service / etc. JSON-LD types                 | Extend `lib/agents/json-ld.ts`; switch on `frontmatter.taxonomy`       |
| Internal links from blog pages to agent pages          | Sanity-side; add a link block component pointing to `/agents/{slug}`   |
| Sitemap-index combining Sanity sitemap + agent sitemap | Add a `pages/api/sitemap-index.xml.ts` or convert to static generation |
| Robots.txt User-agent–specific rules                   | Edit `scripts/generate-robots-txt.ts`                                  |

## Glossary

| Term                            | Meaning                                                                                                                                                     |
| ------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Agent article**               | An AI-targeted markdown derivative of a blog post, stored at `content/agents/{slug}.md`.                                                                    |
| **Agent transformation prompt** | The Claude prompt PublishForge uses to convert blog content into agent markdown. Versioned in PublishForge's `agent_transformation_prompts` table.          |
| **WIKI article**                | A compiled knowledge-base article in PublishForge's Supabase, derived from the document corpus and used to ground agent transformations in factual content. |
| **llms.txt**                    | A site-root file following the [llmstxt.org](https://llmstxt.org/) convention, listing AI-accessible content for crawlers.                                  |
| **sitemap-agents.xml**          | An XML sitemap dedicated to agent pages, kept separate from the Sanity-driven `/api/sitemap`.                                                               |
| **Prebuild generators**         | The three Node scripts at `scripts/generate-*.ts` that run before `next build` to refresh the discovery artifacts.                                          |
| **AGENT_CONTENT_DIRS**          | The single source-of-truth constant at `lib/agents/content-dirs.ts` listing every directory the route + generators should scan.                             |
